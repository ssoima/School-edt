Test Cases
==========
- launchable
  Abgabe kann kompiliert und das Resultat gestartet werden.

- recsmth
  Das Programm sendet irgendwelche Daten.

- denies-short, denies-long, denies-outofrange, denies-inval
  Testet das Parsen der MAC Adresse, aka. wie verhält sich das Programm
  wenn es eine "falsche" MAC bekommt.

- headercorrect
  Ethernet-Header entspricht den Erwartungen.

- payloadcorrect
  WoL-Payload entspricht den Erwartungen.

- reccorrect
  Kombination aus headercorrect und payloadcorrect.


Resultate
=========
- Success
  Abgabe hat den Test erfolgreich bestanden

- Failed
  Abgabe hat den Test nicht bestanden

- Precondition was not statisfied
  Test wurde nicht ausgeführt, da ein hierfür notwendiger Test nicht
  erfolgreich war


Mögliche Gründe für Fehlerhafte Tests (neu 2015-05-09)
======================================================
aka was tun, wenn das paket auf http://wol.net.in.tum.de ankommnt, die
Abgabetests aber Failed oder Precondition was not statisfied reporten.

Der Tester hat genaue Erwartungen daran was das getestete Programm
ausgegeben soll. Für WoL ist die Annahme das nichts ausgegeben wird,
andernfalls wird der Test als Failed klassifiziert. (Hintergrund: bei
Weiteren Programmieraufgaben wird eine Ausgabe erwartet, die dann von
uns geparst wird, das geht vor allem dann gut wenn man genau matchen
kann.)

Eine Fehlerursache können also zusätzliche print Statement auf stdout
sein (so genanntes printf Debugging [1]). 
Dann kann weiter verwendet werden, solang nicht auf stdout sondern
stderr [2] augegeben wird.

Für C also "printf(...)" dann "fprintf(stderr, ...)" verwendet wird.
Und für Java anstelle "System.out.print" dann "System.err.print"
verwendet wird.

[1] https://en.wikipedia.org/wiki/Debugging#printf_debugging
[2] https://en.wikipedia.org/wiki/Standard_streams#Standard_error_.28stderr.29


Anmerkung zur Bewertung
=======================
Erfolgreiche Abgabetests stellen eine Orientierung dar, ob das Programm
den Erwartungen entspricht, sind aber nicht die alleinige Grundlage für
die Bewertung.
Insbesondere gibt es *kein* direktes Mapping von Anzahl erfolgreicher
Tests zu erreichten Punkten.
Weiterhin berücksichtigen wir für die Bewertung auch den abgegebenen
Source Code.
Kriterien, die nicht von den automatischen Abgabetests berücksichtigt
werden können, sind z.B. korrekte Berücksichtigung der Byteorder
unabhängig von der Host Architektur oder graceful Termination ohne
Exceptions.


wol.net.in.tum.de
=================
Die Tests gegen wol.net.in.tum.de (Ergebnis auf
http://wol.net.in.tum.de) sind unabhängig von den Abgabetests und können
ohne Zeitbeschränkungen selbstständig ausgeführt werden.
Es ist natürlich sinnvoll, die Abgabetests erst dann anzufordern, wenn
der selbstständige Test gegen wol.net.in.tum.de erfolgreich war.
